## 1. Verzeichnis + Compose-Datei anlegen

```bash
sudo mkdir -p /opt/claude-os
sudo chown $USER:$USER /opt/claude-os
cd /opt/claude-os
```

`docker-compose.yml` anlegen (vereinfachte v1.7.1-Variante mit Multi-User + Admin-API):

```yaml
services:
  claude-os:
    image: ghcr.io/yannikits/claude-os:v1.7.1
    container_name: claude-os
    restart: unless-stopped
    ports:
      - "127.0.0.1:3000:3000"   # nur localhost — reverse-proxy davor
    volumes:
      - claude-os-data:/data
    environment:
      # ---- Pflicht ----
      CLAUDE_OS_AUTH_TOKEN: ${CLAUDE_OS_AUTH_TOKEN}
      CLAUDE_OS_SECRETS_PASSPHRASE: ${CLAUDE_OS_SECRETS_PASSPHRASE}

      # ---- Multi-User Stage 2 (Email-Login) ----
      # Aktiviert sich automatisch sobald users.sqlite existiert (siehe Schritt 4).

      # ---- Admin-HTTP-API (neu in v1.7.1) ----
      # Kommagetrennt, klein geschrieben. Nur diese Adressen dürfen
      # POST /api/admin/users* aufrufen.
      CLAUDE_OS_ADMIN_EMAILS: ${CLAUDE_OS_ADMIN_EMAILS}

      # ---- Optional ----
      # Session-Persistenz (Re-Login nach Container-Restart vermeiden):
      CLAUDE_OS_SESSION_PERSIST: "1"
      # Self-Registration aktivieren (NUR hinter Cloudflare Access / VPN):
      # CLAUDE_OS_ALLOW_REGISTRATION: "1"

volumes:
  claude-os-data:
```

## 2. Secrets generieren + `.env` befüllen (Setup-Wizard, empfohlen)

```bash
cd /opt/claude-os
curl -fsSL https://raw.githubusercontent.com/yannikits/Claude-portable/main/scripts/setup.sh | bash
```

Der Wizard fragt interaktiv ab:
- **Admin-Email** (für Multi-User + HTTP-Admin-API)
- **Session-Persistenz** (default ja — empfohlen)
- **Self-Registration** (default nein — nur hinter Cloudflare Access aktivieren)

Generiert `.env` mit `chmod 600`, zeigt am Ende die nächsten 3 Befehle.

**Alternative (manuell, ohne Wizard):**
```bash
cat > .env <<EOF
CLAUDE_OS_AUTH_TOKEN=$(openssl rand -hex 32)
CLAUDE_OS_SECRETS_PASSPHRASE=$(openssl rand -hex 32)
CLAUDE_OS_ADMIN_EMAILS=deine@email.com
CLAUDE_OS_SESSION_PERSIST=1
EOF
chmod 600 .env
```

## 3. Erster Container-Start (Single-User Stage 1)

```bash
docker compose pull
docker compose up -d
docker compose logs -f claude-os
```

Erwarteter Log-Output:
```
server: listening at http://0.0.0.0:3000
server: single-user auth (1 token)
```

Stage 2 ist **noch nicht** aktiv — das passiert im nächsten Schritt.

---

## 4. Multi-User Stage 2 aktivieren — ersten Admin anlegen

sql.js ist single-writer-only → Container **stoppen** vor User-Anlage:

```bash
docker compose stop claude-os

docker compose run --rm claude-os \
  node /app/dist/cli/index.js users create \
  --email deine@email.com \
  --password 'mindestens-12-zeichen-passphrase'

docker compose start claude-os
```

Erwarteter Log-Output nach Start:
```
claude-os serve: Multi-User Stage 2 enabled (1 users), sessions persisted
server: admin HTTP API enabled (Web-7-7)
```

**Validierung:** Die Email in `CLAUDE_OS_ADMIN_EMAILS` muss exakt der bei `users create` entsprechen — sonst greift die Admin-Gating nicht.

## 6. Reverse-Proxy davorhängen (TLS + öffentliche Erreichbarkeit)

Drei Optionen ausführlich in `docs/server-deployment.md` Schritt 3. Schnellste Empfehlung wenn Cloudflare schon da ist:

### Option B: Cloudflare-proxied + Origin-Cert (empfohlen)
1. In Cloudflare-Dashboard: SSL-Modus = **Full (strict)**.
2. Origin-Cert generieren (`*.deine-domain.tld`), als `.pem` + `.key` herunterladen.
3. nginx proxy manager: neuer Proxy Host
   - Domain: `claude.deine-domain.tld`
   - Forward to: `<VM-IP>:3000`
   - SSL: Custom-Cert (Origin-Cert hochladen)
   - **WebSockets aktiv** (für `/api/pty/ws` — Chat-Page)
4. DNS-Record: A-Record `claude` → VM-IP, **proxied (orange wolke)**.

### Option C: Cloudflare Tunnel (Zero-Trust, keine offenen Ports)
Detail siehe Vollreferenz; lohnt wenn die VM keine Public-IP haben soll.

## 9. Admin-HTTP-API (neu v1.7.1) testen

Cookie + CSRF holen via Browser-Login. Dann z.B. via `curl`:

```bash
# Im Browser DevTools: Cookies "claude_os_session" + "claude_os_csrf" kopieren.
SESSION="..."
CSRF="..."

# User listen
curl -s "https://claude.deine-domain.tld/api/admin/users" \
  -H "Cookie: claude_os_session=$SESSION; claude_os_csrf=$CSRF"

# Neuen User anlegen
curl -s -X POST "https://claude.deine-domain.tld/api/admin/users" \
  -H "Cookie: claude_os_session=$SESSION; claude_os_csrf=$CSRF" \
  -H "x-csrf-token: $CSRF" \
  -H "Content-Type: application/json" \
  -d '{"email":"kollege@firma.de","password":"strong-password-12+"}'

# User disablen (revoked auch alle laufenden Sessions des Users)
curl -s -X POST "https://claude.deine-domain.tld/api/admin/users/kollege@firma.de/disable" \
  -H "Cookie: claude_os_session=$SESSION; claude_os_csrf=$CSRF" \
  -H "x-csrf-token: $CSRF"
```

Endpoint-Übersicht:
| Method | Path | Effect |
|--------|------|--------|
| GET    | `/api/admin/users` | Liste (incl. disabled) |
| POST   | `/api/admin/users` | Create (201/409/400) |
| POST   | `/api/admin/users/:idOrEmail/disable` | flip + revoke sessions |
| POST   | `/api/admin/users/:idOrEmail/enable`  | flip |
| POST   | `/api/admin/users/:idOrEmail/reset-password` | new password + revoke sessions |

## 10. Backup

```bash
# Volume-Pfad finden
docker volume inspect claude-os-data --format '{{ .Mountpoint }}'

# Periodisch sichern (z.B. via cron + rsync ins NAS)
sudo rsync -av /var/lib/docker/volumes/claude-os-data/_data/ /mnt/backup/claude-os/
```

**Was im Volume liegt (kritisch zum Backup):**
- `vault/` — alle Notes + Skills + workspaces
- `data/users.sqlite` — Email-Login-Stammdaten
- `data/sessions.sqlite` — aktive Sessions (verlierbar — re-login möglich)
- `data/audit/` — JSONL-Audit-Trail
- `~/.claude/` (im Container) — Anthropic-Login-Token

Zusätzlich Proxmox-Snapshot vor jedem Update — Application-konsistent reicht für Claude-OS.

## 12. Troubleshooting-Kurzcheck

| Symptom | Erste Maßnahme |
|---------|----------------|
| `$CLAUDE_OS_AUTH_TOKEN is required` | `.env`-Datei prüfen, `docker compose config` läuft? |
| Login schlägt 401 fehl | User wirklich enabled? `docker exec claude-os node /app/dist/cli/index.js users list` |
| CSRF-403 auf POST | `x-csrf-token`-Header gesetzt + Wert == `claude_os_csrf`-Cookie? |
| Admin-API 403 | Email in `CLAUDE_OS_ADMIN_EMAILS` exakt + lowercased? Restart nach Änderung? |
| SSE bricht nach 100s ab | Cloudflare-proxied? `sseHeartbeatMs` < 100000 setzen oder Cloudflare-Caching-Settings für `/api/events`-Pfad aus |
| WebSocket auf `/api/pty/ws` failt | Reverse-Proxy WebSockets aktiviert? Bei nginx PM: Tickbox „Websockets Support" am Proxy-Host |

Vollständige Troubleshooting-Sektion: `docs/server-deployment.md` §Troubleshooting.

## Anhang — wichtige Env-Vars Referenz

| Env-Var | Default | Effect |
|---------|---------|--------|
| `CLAUDE_OS_AUTH_TOKEN` | _required_ | Bearer-Token (Stage 1 Fallback) |
| `CLAUDE_OS_SECRETS_PASSPHRASE` | _required_ | Masterkey für encrypted-file Secrets |
| `CLAUDE_OS_ADMIN_EMAILS` | _empty_ | CSV-Allowlist für Admin-HTTP-API (sonst Routes nicht registriert) |
| `CLAUDE_OS_ALLOW_REGISTRATION` | `0` | `1` = `/api/auth/register` aktiv (Cloudflare-Access davor!) |
| `CLAUDE_OS_SESSION_PERSIST` | `0` | `1` = sessions.sqlite, Re-Login nach Restart vermeiden |
| `CLAUDE_OS_INSECURE_COOKIES` | `0` | `1` = `Secure`-Flag drop (NUR Dev/localhost) |
| `CLAUDE_OS_MULTI_USER` | _auto_ | `1` forciert Stage 2 (auch ohne users.sqlite) |
| `CLAUDE_OS_DISABLE_MULTI_USER` | _empty_ | `1` forciert zurück auf Stage 1 |
| `CLAUDE_OS_SECRETS_BACKEND` | _container default_ | bleibt `encrypted-file` im Container |
| `CLAUDE_OS_STATIC_DIR` | `/app/gui/dist` | Vite-Build im Image |
| `CLAUDE_OS_LOG_LEVEL` | `info` | `debug` für mehr Detail |
